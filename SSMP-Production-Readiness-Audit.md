# SSMP — Production Readiness Audit

**Reviewed:** `ssmp-main.zip` snapshot, June 2026
**Scope:** `apps/api` (Node/Express/Postgres), `apps/admin` (React/Vite), `apps/public` (Next.js), `apps/android` (Kotlin/Compose), `packages/shared-types`, plus all docs and migrations.

## How to read this report

This was built by tracing real request paths end to end — not just reading files in isolation. Where a finding says a specific call will fail, it's because the route was located, its middleware chain read, and the calling client's code checked against it, not inferred from style alone. Areas not directly exercised this pass (full `shared-types/index.ts`, full Android Room/WorkManager sync code, every admin component's UI behavior, accessibility, load/performance under real traffic) are called out explicitly rather than assumed fine — treat those as unaudited, not passed.

The headline finding across almost every section: **the architecture and code quality are genuinely strong — better than most early-stage builds.** The problems are not "the code is sloppy." They're "several pieces were built correctly in isolation but were never connected to each other," and "one foundational piece (real user accounts) was never built at all." That's a very fixable shape of problem, and the fixes are concrete and listed below.

## 1. Critical Findings (fix before any real user touches this)

### 1.1 There is no real authentication system anywhere in the stack
The `users` table has a `password_hash` column and the RBAC permission matrix has full `user:*` permissions — the schema was designed for real accounts. But no module implements `/api/auth/login`, `/api/auth/register`, or any user-management routes. The only way to get a JWT today is `POST /api/auth/dev-token`, which mints a token for any role with no credential check, and is only blocked from production by a single `NODE_ENV === 'development'` check in `app.js`.

This isn't a one-off — it surfaces independently in all three clients:
- The Admin Portal's `socket.ts` calls the dev-token endpoint directly as its only auth path.
- The Android app's `SsmpApiService.kt` calls `POST /api/auth/login` and `POST /api/auth/register` — **neither endpoint exists on the backend.** Those calls will 404 today.

**Why it matters:** until this exists, nobody can actually onboard a real Coach, Registrar, or Competition Admin. Everything downstream (ownership checks, approvals, audit trails) assumes a real `users.id` that nothing currently creates.

### 1.2 The Admin Portal is not connected to the real backend for ~90% of its screens
`apps/admin/src/App.tsx` seeds and reads everything from an in-memory `mockDb` — Competition Wizard, Registration Queue, Roster Queue, Fixture List, Transfers/Discipline, Media Gallery, News, Audit Log, and the Dashboard all run on fake data that resets on page reload and has no relationship to Postgres. Only the match-control screen (`OfficialsMatchEvents.tsx`, via `matchApi.ts`/`socket.ts`) talks to the real API. `AGENTS.md` self-documents why: *"The admin app was generated by Google AI Studio. Component logic is untouched; only imports were fixed."* — the import swap from the playbook's merge step happened, but the data-fetching layer underneath was never rebuilt.

**Why it matters:** as it stands, approving a registration or a roster in the admin UI does nothing to the real team/player records. This is the single highest-leverage fix available — every other admin feature is otherwise complete-looking.

### 1.3 No standings endpoint exists anywhere — on either the backend or any client
`match.service.js` correctly computes and persists standings into the `standings` table on publish. But there is no `/api/standings` route in `app.js` or any module. Both consumers independently assumed it would exist and both are stubbed:
- The public site's standings page hardcodes every row to `played: 0, won: 0, drawn: 0, lost: 0, points: 0` — it never calls a standings endpoint at all.
- The Android app's `SsmpApiService.kt` calls `GET /api/competitions/{id}/standings`, which also doesn't exist — this 404s.

**Why it matters:** standings is a headline PRD feature for both the public site and the team app, and the data already exists correctly in Postgres — only the read endpoint is missing. This is a small, contained fix with disproportionate value.

### 1.4 Lineup submission has no real persistence layer
The match lifecycle correctly includes `lineups_submitted`/`lineups_locked` states, and `permissions.js` defines `lineup:submit`/`lineup:read`/`lineup:lock` — but no `lineups` table exists, no lineup module/routes exist, and nothing enforces "a suspended or injured player cannot be selected" anywhere server-side. The Android client improvised around this gap: `getLineups`/`submitLineup` are aliased to the match-events endpoints (`GET/POST /api/matches/{id}/events`), meaning a submitted lineup is stored as a generic match event with no defined event type for it, which the discipline engine and match-event validation schema weren't designed to handle.

**Why it matters:** this is one of the five core Android screens and a named discipline-engine integration point in the PRD. It currently has no real backend behind it in any client.

### 1.5 The public site cannot load almost any of its core data against the real API
Every route is `auth`-protected except `/api/public/matches/*` (verified across every `*.routes.js` file — this is the only public-readable surface in the entire API). The public site's `getCompetitions()`, `getTeams()`, `getPlayers()`, `getOrganizations()`, `getSeasons()`, and `getMedia()` all call the authenticated routes directly with no Authorization header. Every one of these will return 401 against the real backend. Only match/score data (which correctly uses `/api/public/matches`) will actually load.

**Why it matters:** the entire premise of the public site — per the PRD — is anonymous, no-login browsing. Right now only live scores work that way; everything else needs either a public read variant on the backend (mirroring the `match-public.routes.js` pattern already established) or the site needs its own service-level credential, which the PRD's "no login" framing argues against.

### 1.6 JWT secret has an insecure, predictable production fallback
`config/env.js`: `jwtSecret: process.env.JWT_SECRET || 'dev-secret-change-in-production'`. If `JWT_SECRET` is ever left unset in a real deployment — a one-line `.env` mistake — every token becomes forgeable using a string that's sitting in the public repo. There's no startup check that refuses to boot without a real secret in production.

### 1.7 The audit trail silently fails for every write made via the current dev-auth flow
`audit_logs.user_id` is `NOT NULL REFERENCES users(id)`, but dev-token IDs (e.g. `"coach-001"`) aren't real UUIDs and don't correspond to any seeded user — `seed.sql` creates zero rows in `users`. `createAuditLog()` wraps its insert in a try/catch that swallows the error and only `console.error`s it — so the *primary* action (creating a team, approving a roster, etc.) still succeeds, but **every audit log entry for every authenticated action silently fails to write** against a real Postgres instance. The integration test suite can't catch this because it runs against an in-memory mock DB that doesn't enforce foreign keys or UUID typing — tests pass while this would fail in real use.

## 2. High-Priority Findings (real bugs, not blocking but should land soon)

### 2.1 Inconsistent row-level ownership checks — a recurring pattern, not an isolated bug
RBAC role-checking is implemented carefully and consistently (`rbac('permission:key')` everywhere). But *ownership* scoping is inconsistently applied on top of it:
- `player.service.js#update` correctly checks `team.coach_id === userId` before letting a Coach edit a player. Good.
- `team.service.js#update` has **no equivalent check** — any authenticated Coach can `PATCH` any team's profile, logo, or branding, not just their own, even though `team.service.js#list` correctly scopes by `coach_id` for the same role. The `isTeamCoach` middleware exists in `rbac.js` for exactly this purpose but is never imported or used in `team.routes.js`.
- Officials have no equivalent check at all: `match:update-status`, `match:submit-report`, and `match:record-event` are gated only by role, not by `match.official_id === req.user.id`. The PRD explicitly states official actions should be "limited to matches they're assigned to (server-enforced)" — this isn't currently enforced anywhere.

**Fix shape:** generalize the pattern already proven in `player.service.js` into a small reusable ownership check, and apply it to team updates and official-scoped match actions.

### 2.2 No jersey-number uniqueness enforcement
`CreatePlayerSchema` validates jersey number range (0–99) but nothing — not the Zod schema, not `player.service.js`, not a DB constraint — prevents two players on the same team from sharing a jersey number. This was an explicit requirement in the original (pre-rebuild) spec and appears to have been dropped during the rewrite.

### 2.3 Editing a player doesn't revert an already-approved roster back to "pending"
`team.roster_approval_status` is a single status on the team. A Coach can add or edit a player at any point inside the registration window — including after a Registrar has already approved the roster — with nothing resetting that approval back to "pending" for re-review. A Registrar could be looking at a stale "approved" status that no longer reflects the actual current roster.

### 2.4 No baseline Express hardening
No `helmet`, no rate limiting anywhere (including on the auth-adjacent dev-token endpoint), `cors()` with no origin restriction, and Socket.IO's CORS defaults to `*`. None of this is unusual for this stage of a build, but all of it needs to land before going live with real users — especially rate limiting, given the dev-token route's risk profile if `NODE_ENV` is ever misconfigured.

### 2.5 Once a match is published, there's no correction path
`VALID_TRANSITIONS.published = []` — fully terminal, by design, for data integrity. But there's also no defined way to correct a score that was wrongly verified and published before an error was caught. Worth a deliberate decision (e.g. a `corrected` sub-flow requiring System Admin) rather than leaving it as a dead end.

## 3. What's Actually Built Well

It's worth being specific about this, since a "be highly critical" audit can read as only-negative otherwise:

- **The vertical-slice module structure is followed faithfully and consistently** across all 12+ entities — routes/controller/service split is uniform, easy to navigate, matches `SSMP-Project-Structure.md` exactly.
- **RBAC is genuinely well-designed**: granular permission keys (`team:approve-roster`, not just "admin can do anything"), defined once in `permissions.js`, referenced everywhere — this is a better pattern than most early-stage admin tools bother with.
- **SQL is parameterized throughout** — no string-concatenation injection risk found anywhere across teams, players, matches, competitions, fixtures.
- **The discipline engine is correctly built**: per-competition configurable thresholds, immediate red-card suspension, auto-serve on match completion, leaderboard — this matches PRD section 6.6 closely and is one of the most complete pieces in the system.
- **Database schema is well-normalized** with real constraints (`CHECK`, `UNIQUE`, FK cascading concerns considered) — not just loose JSONB blobs everywhere.
- **The match lifecycle state machine** (`VALID_TRANSITIONS`) correctly encodes the 17-state flow from the PRD and rejects invalid transitions.
- **The public site and Android app both correctly import from `@ssmp/shared-types`** rather than redefining types — the core architectural goal of the shared-contract layer is actually working as intended in the pieces that are wired up.
- **`AGENTS.md` is excellent** — accurate, current, and self-aware about exactly the kind of gaps this audit found (it explicitly flags the admin app's mock-data status and the dev-only auth). Whoever maintained it was tracking reality honestly, which made this review faster and more accurate.

## 4. Real-World Simulation: running an actual season today

Walking through the PRD's own lifecycle example against what's built:

| Step | What happens today |
|---|---|
| Admin sets up organization/season/competition | Works — real backend, real persistence |
| Schools register | No real route is wired for this on the admin side (mock data); backend route exists and works if called directly |
| Registrar approves registrations | Admin UI approval doesn't touch the real team record (1.2) |
| Coach logs in | Cannot — no login exists (1.1) |
| Coach builds roster | Backend correctly enforces registration window + ownership; works if called directly, not yet reachable through a real login flow |
| Registrar approves roster | Same mock-data issue as registrations |
| Fixtures generated | Backend logic is solid; admin UI managing it is mock-only |
| Match day: official runs match control | **This works end-to-end against the real API** — the one fully-wired admin workflow |
| Coach submits lineup | No real persistence (1.4) |
| Standings update after a result | Backend computes it correctly; nothing can read it back (1.3) |
| Public follows along | Live scores work; everything else 401s (1.5) |
| Season ends, archive | Untested this pass — `season:archive` route exists, not exercised |

The match-control day-of-game flow is the most production-ready single workflow in the system. Everything upstream of match day (registration → roster → approval) and downstream (standings, public visibility) has at least one missing connection.

## 5. Missing Features, Prioritized

**Critical** (blocks first real use)
- Real user registration/login (1.1)
- Wire Admin Portal screens to the real API instead of `mockDb` (1.2)
- `/api/standings` endpoint + wire both clients to it (1.3)
- Real `lineups` table + endpoints + suspended/injured-player enforcement (1.4)
- Public read-access strategy for competitions/teams/players/media (1.5)

**Important**
- Ownership checks for team updates and official-scoped match actions (2.1)
- Jersey number uniqueness (2.2)
- Roster re-review trigger on post-approval edits (2.3)
- Helmet + rate limiting + CORS allowlist (2.4)
- Score-correction path post-publish (2.5)
- Production startup check that refuses to boot without a real `JWT_SECRET` (1.6)

**Nice to have**
- Integration tests that run against a real (containerized) Postgres for at least the write paths that touch `audit_logs`, to catch the class of bug in 1.7 that the mock DB structurally cannot catch
- Reconcile the `@types/react` workspace-level override (pinned to 18.x) against Admin's own React 19 devDependency — currently inconsistent, not yet causing visible breakage

## 6. Recommended Next Prompts

Given how the rest of this build has gone, here are ready-to-use Claude Code prompts for the top three critical items — in priority order:

```
Build a real authentication system for apps/api: POST /api/auth/register
and POST /api/auth/login, using bcrypt for password hashing against the
existing users table. Issue the same JWT shape the dev-token endpoint
currently issues (id, email, role) so nothing downstream needs to change.
Keep the dev-token endpoint for local testing, but add a startup check in
config/env.js that throws if NODE_ENV is "production" and JWT_SECRET is
unset or equals the current default fallback string.
```

```
Add a GET /api/standings/:competitionId endpoint (and a public variant at
/api/public/standings/:competitionId, unauthenticated) that reads from the
standings table already maintained by match.service.js. Wire the public
site's standings page and the Android app's getStandings call to use it
instead of hardcoded zeros / a 404.
```

```
Rewrite apps/admin/src/App.tsx and every feature component that currently
imports mockDb (CompetitionWizard, RegistrationQueue, RosterQueue,
FixtureList, TransfersDiscipline, NewsMedia, MediaGallery, the Dashboard)
to fetch from and write to the real API at apps/api instead, following the
same pattern already working in matchApi.ts. Keep mockDb.ts only as a
fallback for when no API URL is configured, clearly labeled as such in the
UI.
```

Android's Room/WorkManager offline-sync implementation (`OfflineQueue.kt`, `SyncWorker.kt`) was located but not read in depth. The `organizations`/`seasons` modules and `transfer.service.js`/`media.service.js`/`notification.service.js` were sampled at the routes level but not read in full. Treat those as open, not cleared.
